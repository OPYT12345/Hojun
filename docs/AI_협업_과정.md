# AI 협업 과정 문서

**프로젝트명:** NFC 기반 스마트 수업 관리 시스템  
**협업 AI:** Claude (Anthropic)  
**협업 기간:** 2026년 3월 ~ 4월  

---

## 1. 협업 방식 개요

본 프로젝트는 **기획자(인간) + 구현 파트너(AI)** 모델로 개발되었다.

- **기획자 역할 (개발자):** 문제 정의, 기능 아이디어 제안, 결과물 검토, 테스트 및 피드백
- **AI 역할 (Claude):** 아키텍처 설계, 코드 구현, 보안 검토, 버그 수정, 최적화 제안

AI는 단순한 코드 생성 도구가 아니라, 설계 단계부터 함께 참여하는 **협업 파트너**로 활용하였다.

---

## 2. 단계별 협업 내역

### Phase 1 — 기반 구조 설계
**기획자 요청:** 로그인/회원가입 시스템, 강사/학생 역할 분리, HTTPS 적용  
**AI 기여:**
- Spring Boot + JPA + MySQL 프로젝트 구조 설계
- BCrypt 비밀번호 암호화, 세션 기반 인증 구현
- HTTPS keystore 설정 방법 안내 및 적용
- 로그인 실패 횟수 제한(LoginAttemptService) 설계

---

### Phase 2 — NFC 출석 시스템
**기획자 요청:** NFC 태그 하나로 출석 처리가 되면 좋겠다  
**AI 기여:**
- `/nfc/{lectureId}` 단축 URL 설계 제안
- `NfcRedirectController` 구현
- 미로그인 상태 접근 처리 로직 설계
- 모바일 접속을 위한 방화벽 오픈 스크립트(`open-firewall.bat`) 제공

---

### Phase 3 — 과제 시스템 설계
**기획자 요청:** 문제 방식보다 자유 서술 과제 방식이 더 적합할 것 같다  
**AI 기여:**
- `Assignment` 엔티티 재설계 (기간 설정, 첨부자료 JSON 구조)
- `Submission` 엔티티 설계 (텍스트 + 파일/링크 복합 제출)
- D-day 카운트다운, 기한 초과 처리 로직
- 강사 제출 현황 15초 자동 갱신 구현

**협업 포인트:** "Problem → Assignment 전환"이라는 기획 변경을 AI가 기존 코드와의 충돌 없이 안전하게 리팩토링

---

### Phase 4 — AI 튜터 / 분석 기능
**기획자 요청:** 학생용 AI 튜터와 강사용 분석 기능을 넣고 싶다  
**AI 기여:**
- OpenAI GPT-4o-mini API 연동 설계
- `AiService` 구현 (학생 튜터, 강사 분석, Q&A 답변 제안)
- 환경변수 기반 API 키 관리 방법 설계 (보안)
- 토큰 낭비 방지를 위한 프롬프트 최적화

---

### Phase 5 — 채점 및 알림 시스템
**기획자 요청:** 채점 결과를 학생이 바로 알 수 있게 해줘  
**AI 기여:**
- `Notification` 엔티티 및 인앱 알림 시스템 전체 설계
- 알림 유형 3종(GRADE / QA_ANSWER / BROADCAST) 정의
- 벨 아이콘 UI, 미읽음 배지, 모달 목록 구현
- `@PreUpdate` 충돌 버그 발견 및 수정

---

### Phase 6 — 공지 방송 시스템
**기획자 요청:** 강사가 전체 수강생에게 공지를 보낼 수 있으면 좋겠어  
**AI 기여:**
- `Announcement` 엔티티 설계
- 수강생 전원 동시 알림 발송 로직 (N+1 방지 쿼리 포함)
- 강사/학생 양측 UI 탭 추가

---

### Phase 7 — Docker 배포 환경
**기획자 요청:** 심사관이 한 번에 실행할 수 있게 해줘  
**AI 기여:**
- 멀티스테이지 `Dockerfile` 설계 (Maven 빌드 → JRE 실행)
- `docker-compose.yml` 설계 (MySQL + Spring Boot 통합)
- `data.sql` 샘플 데이터 자동 삽입 구조
- `start.bat` 로컬 실행 스크립트

---

### Phase 8 — 보안 검토
**AI 주도적 기여 (기획자 요청 없이 발견):**
- SSE 엔드포인트 강사 인증 누락 → 수정
- 파일 업로드 경로 순회(Path Traversal) 취약점 → 수정
- 강의 소유권 미검증 → 수정
- 잘못된 입력값에 500 반환 → 400으로 수정

## 3. AI 협업의 핵심 원칙

### 3-1. 기획자가 방향을 결정, AI가 구현을 담당
모든 기능의 **존재 여부는 기획자가 결정**하였다. AI는 "어떻게 구현할 것인가"를 담당하되, 기획자의 승인 없이 임의로 기능을 추가하지 않았다.

### 3-2. AI의 제안을 기획자가 검토
AI가 구조나 방식을 제안할 때, 기획자가 검토 후 채택 여부를 결정하였다. 예: 도메인 주도 패키지 구조 전환은 AI가 제안했지만 현재 규모에서는 불필요하다고 판단하여 보류.

### 3-3. 테스트는 기획자가 직접
기능 구현 후 실제 브라우저/모바일에서의 테스트는 기획자가 직접 수행하고, 발견된 버그를 AI에게 전달하여 수정하였다.

### 3-4. 보안 및 안정성은 AI가 주도적으로 검토
배포 전 코드 검토 단계에서 AI가 실제 운영 환경에서 문제가 될 수 있는 보안 취약점을 능동적으로 발견하고 수정하였다.
